# 02 — Billing

Billing is where the SOV becomes money the owner owes. The design goal (from the
founder review): billing must be **accurate, intuitive, sequential, and it must
remember** — the biller should never hop around or re-key last month's numbers.

## The pay-application builder (the spine of the bill)

A billing period produces a **pay application** (a.k.a. requisition), stored in
`public.billing_applications`. The builder is a guided, always-visible sequence
— every step is either actionable or disabled-with-a-reason, never missing:

```
1. Format      → Client invoice  OR  AIA G702/G703
2. SOV         → import the schedule of values (the contract lines you bill)
3. This period → enter percent-complete / stored materials per line
4. Generate    → finalize, then download AND email the package
```

Code: [`AiaApplicationStepper.tsx`](../../src/components/billing/AiaApplicationStepper.tsx)
(the stepper UI, gating, and out-of-sequence routing) and
[`aia-builder-steps.ts`](../../src/lib/aia-builder-steps.ts) (the step/gate
logic). An out-of-sequence "Generate" click routes the biller to the blocking
step instead of silently doing nothing. Generating with overbilled lines
requires one explicit confirm (the overbilling guard-rail).

## The carry-forward memory (this is real, and it's the trust anchor)

The founder was most skeptical that the app truly "remembers" the prior bill. It
does, and correctly. When a new pay application is created, the engine pulls the
**previous** application (`apps[i-1]`) and carries its to-date figures into this
application's "from previous" columns — proper AIA G703 columns D/E/F/G.

- Logic: [`billing.functions.ts`](../../src/lib/billing.functions.ts) (new app
  pulls the prior app and sets `work_completed_previous` = prior app's
  to-date), [`billing-line-generation.ts`](../../src/lib/billing-line-generation.ts),
  and the G703 math in [`aia-math.ts`](../../src/lib/aia-math.ts).
- Surfaced in the UI as a **"Carried forward from {prior app}: {amount} already
  certified…"** banner on the this-period screen
  ([`BillingEnhancements.tsx`](../../src/components/billing/BillingEnhancements.tsx)),
  so the memory is unmistakable and trusted.

`amount_billed` on a billing application is the **per-application** amount (this
period), not a cumulative — the portfolio total is the sum across applications.
The running billed-to-date is therefore a cumulative sum in submission order
(this matters for the Billing history report).

## AIA G702 / G703

For companies that bill AIA (owners, lenders, bonding agents expect it), the
package is a lender-grade G702 (application/certificate) + G703 (continuation
sheet). The PDF is generated by [`aia-pdf.ts`](../../src/lib/aia-pdf.ts) with
`pdf-lib`. Formatting was hardened (G703 vertical grid lines, full column
headers, no logo/prep-date overlap) — to inspect it, a node fixture renders the
real package to a file (`npm run test:billing:aia`, see
[`scripts/aia-pdf-fixture-smoke.ts`](../../scripts/aia-pdf-fixture-smoke.ts)).
Companies that never choose AIA never see AIA affordances beyond the format
toggle.

The package can be **downloaded** or **emailed** to the client from the generate
step. Email goes to **all** billing contacts with `can_view_billing` (deduped),
via the transactional email pipeline (below).

**Downloading goes through the one safe path,
[`download-file.ts`](../../src/lib/download-file.ts) — never hand-roll a blob
download.** A field bug had the AIA package silently fail to download in
Safari/iOS: revoking the blob URL synchronously after `anchor.click()` races the
browser's async fetch of that blob (Chrome tolerates it; Safari cancels the
download with no error). Every PDF/CSV/text download now delegates to
`download-file.ts`, which revokes on a 60-second delay. If you add a download,
call `downloadFileBytes` / `downloadTextFile` — a phase0 pin guards against a
"cleanup" reintroducing the synchronous revoke.

## Retainage

Retainage is the portion the owner withholds until closeout. It is tracked per
line and per application (`retainage`, `total_retainage_held`,
`retainage_released_this_period` on the application; `retainage_pct` defaults
per project, 10% fallback). The **net retainage held** per job is computed by
the WIP engine (`total_retainage_net`, see
[03](03-wip-schedule-and-reports.md)) and is the figure the Retainage report and
the WIP report both use — so they agree.

## Change orders in the billing flow

Approved change orders carry into billing with their own billed percent (G702
line 2). They are allocated to cost codes (`change_order_allocations`) so the
contract growth reaches the SOV lines. See
[01 — change orders](01-the-spine-sov-and-budget.md#change-orders-grow-the-contract).

## Invoices & payments

- `public.billing_invoices` — the client-facing invoice/requisition record
  (total due, paid amount, status, issue/due dates).
- `public.payment_ledger` — recorded payments (amount, status, paid_at). The
  cents-exact reconciliation between invoices and the ledger lives in
  [`payments-domain.ts`](../../src/lib/payments-domain.ts)
  (`reconcileInvoiceFromLedger`, refund reversal, pending-payment locks). Stripe
  is the processor for direct charges and ACH; Overwatch records and reconciles.
- Money is **integer cents** end to end; display uses `fmtUSDCents` from
  [`billing-format.ts`](../../src/lib/billing-format.ts) (note: the name means
  "renders cents precision" — it takes a **dollar** value and shows 2 decimals;
  it does **not** divide by 100).

## Transactional email

Billing emails (invoice notifications, the AIA package link, magic links,
invites) go through `sendTransactionalEmail` in
[`src/lib/email/send.ts`](../../src/lib/email/send.ts) →
`POST /lovable/email/transactional/send` → a React Email template → an
`enqueue_email` RPC → the provider. Templates live in the email registry
(`invoice-notification`, `ior-report-notification`, etc.). Sends are idempotent
(idempotency key) and never block a billing action.

## The biller's home (Workspace A)

The receivables cockpit is the biller's landing surface — outstanding
receivables, aging, next billing dates, per-project entry points. It lives in
[`ReceivablesCockpit.tsx`](../../src/components/billing/ReceivablesCockpit.tsx)
and the portfolio billing route. In the design language it is "Workspace A"; the
PM's "Workspace B" is [Daily WIP](04-daily-wip-workspace-b.md).
